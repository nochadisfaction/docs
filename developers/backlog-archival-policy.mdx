---
title: 'Backlog Archival Policy'
description: 'Operational rules for deciding when to merge, archive, restore, or defer idea-level work.'
---

## Purpose

This policy keeps the ideas backlog usable while preserving recoverable history.
It applies to issue-level work in Beads unless a more specific domain policy already
exists.

## Decision rules

Use the following sequence for each open idea:

1. If a new submission has the same intent, owner, and expected outcome as an
existing item, merge it into one canonical issue.
2. If it is clear evidence that the idea is no longer relevant, not feasible, or
a lower-priority duplicate, archive it.
3. If it is valuable but waiting on timing, capacity, or external dependency, defer
it.
4. If it is still active, keep it open and route to the normal workflow.

### Merge

Merge when all of these are true:

- The goals are materially the same and execution would duplicate work.
- The older item contains context worth preserving (research notes, dependencies, or
assignee history).
- You can keep one canonical home that has the best context.

When merging:

- Keep the richest issue as the canonical destination.
- Close the duplicate with `bd supersede` and keep a clear pointer.
- Add a short note before closing if any context in the source is not copied.
- Transfer any actionable follow-up into the canonical issue.

```bash
bd supersede pixelated-old --with pixelated-new
bd comment pixelated-old "Merged into pixelated-new for canonical execution."
```

### Archive

Archive when:

- The idea is clearly outdated or no longer aligned to current strategy.
- The idea is a low-signal duplicate without new evidence.
- The idea was invalid, unsafe, or impossible without major rework.

Archive is a reversible state that keeps history available:

- Set a backlog state marker so operations can find it later.
- Keep the issue closed so active queues stay focused.

```bash
bd set-state pixelated-xxx lifecycle=archived --reason "No longer relevant after roadmap update"
bd close pixelated-xxx --reason "Archived by policy: no longer aligned"
```

### Restore

Restore when one or more conditions reactivates value:

- New customer or internal evidence shows concrete demand.
- An architecturally dependent dependency is now in place.
- A team previously blocked by policy, budget, or regulatory issues now has a path.

Restore is a two-step action:

- Reopen the issue.
- Remove the archived lifecycle state marker.

```bash
bd reopen pixelated-xxx
bd set-state pixelated-xxx lifecycle=active --reason "Restored due to new supporting evidence"
```

### Defer

Use defer for ideas that are valid but intentionally not in active execution.

- Use `bd defer` when the item has no immediate execution slot.
- Add a revisit date so it comes back automatically.

```bash
bd defer pixelated-xxx --until=+4w
```

## Operational cleanup workflow

Run this workflow weekly to keep the backlog usable.

### 1) Triage view

- Pull open ideas and identify those with no activity.
- Pull open items that have been stale for 30+ days.

```bash
bd list --status open --no-parent
bd stale --status open --days 30 --limit 200
```

### 2) Apply rules consistently

For each item in the triage set:

- Merge duplicates immediately into canonical issues.
- Archive items that fail both current relevance and feasibility checks.
- Defer items that are valid but not ready.
- Keep active items as open, with updated notes and next owner.

### 3) Capture decisions

Every non-active decision should include a short comment:

- Why it was merged, archived, deferred, or restored.
- Trigger event (new evidence, roadmap change, dependency resolved).
- Follow-up action or owner when needed.

Recommended comment format:

```text
Decision: archive
Reason: duplicate evidence + no new value after roadmap review
Next check: 2026-07-01
```

## Follow-up work

No Jira or product documentation changes are required to execute this policy.
The only integration needed is operator discipline:

- Use the above workflow in the Beads triage process.
- Ensure future "idea", "task", and "spike" intake follows these same rules.
